# Fall / gate nearby helper follow-up

`play_movement_state_sound` xrefs:

- `0x43d143` | `update_subgoldy`
- `0x43d19e` | `update_subgoldy`

Those callsites belong to the movement-progress branch, not the attachment-exit threshold branch.

Recovered attachment-exit threshold behavior from `update_subgoldy`:

- `0x43ce8a` / `0x43ce90`
  - `attachment_exit_progress += attachment_exit_progress_step`

- `0x43ce9c` .. `0x43cec8`
  - once `attachment_exit_progress > 0.7`, `attachment_exit_gate_a` gates a one-shot voice trigger

- `0x43ced3` .. `0x43ceef`
  - if `player + 0x2d8 == 0` and `world_y < -6`, that same first gate also dispatches cutscene animation `5`

- `0x43cef4` .. `0x43cf25`
  - `world_y < -7` is a separate deeper threshold:
    - plays a second one-shot voice
    - sets `attachment_exit_gate_b = 1`
    - also forces `attachment_exit_gate_a = 1`

- `0x43cfc1` .. `0x43cfe2`
  - `attachment_exit_pending != 0` still zeroes `progress_timer`, so this branch is part of the in-fall transition rather than normal movement progression

- separate older death path:
  - elsewhere in `update_subgoldy`, `world_y < -7 && death_active == 0` still calls `initialize_subgoldy_death`

Nuke cluster meaning:

- `0x447110` `initialize_nuke` seeds 25 sprites, `owner_y - 5`, a phase step of `2 * dt`, then tail-calls `update_nuke`
- `0x4471e0` `update_nuke` orbits the 25 sprites at radius `7.0`
- `0x4470e0` `uninit_nuke` kills the 25 sprites and clears state

Conclusion:

- The `-6 / -7` thresholds do not settle respawn or final-loss timing.
- They gate one-shot attachment-exit voice/cutscene effects inside the fall transition.
- The `0x4470e0 / 0x447110 / 0x4471e0` cluster is unrelated to that logic.
